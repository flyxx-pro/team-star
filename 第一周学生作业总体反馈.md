# 第一周学生作业总体反馈

日期：2026-06-06

第一周大家都很努力，也都在尝试把“下载招股说明书、解析 PDF、定位相关章节、抽取结构化 JSON、做日志和校验”这条链路跑起来，这一点很重要。但从目前结果看，距离圆满完成任务要求还有明显距离。这个项目的难点不只是写代码，也不是把 LLM 接上去就结束，而是要让每一个抽取结果都能被追溯、被复核、被别人复现。

下面的反馈先总结第一周共同优点和共同问题，再按 GitHub 提交给出具体建议，最后说明第二周重新安排。

## 一、大家做得好的地方

### 1. 已经基本理解了任务的完整流程

多数同学不只是交了几个 JSON 文件，而是尝试搭建从 PDF 到结构化结果的处理流程，包括下载、解析、候选文本定位、抽取、日志记录和结果输出。对第一周来说，这是一个好的起点。

这个项目后续不是比谁一次抽得最多，而是比谁能把流程稳定下来，让别人拿到同样 PDF 后也能跑出同样或接近的结果。

### 2. 大家都开始重视“证据”

不少结果中已经包含 `evidence_text`，这说明大家意识到招股说明书抽取不能只给结论，还要告诉别人“这个结论来自哪段原文”。这是后续能否形成可信数据集的基础。

尤其是刘宇轩、苏宇垚组和李泽润组，证据链相对较好；赵秉清组虽然部分证据更像人工概括，但候选文本和复盘材料比较充分，也体现了证据意识；霍泓锟提交中也有候选文本和证据页码，说明已经开始注意“结果要能回到原文”。

### 3. 有些组已经有了工程化意识

李泽润组的日志、校验和相对路径设计比较完整；刘宇轩、苏宇垚组提交了明确的 Deepseek 抽取提示词；赵秉清组对流程、失败样本和异常情况有较多记录；霍泓锟提交了分步骤代码和人工复核清单；杨苗鑫和尚恬组也搭出了下载、解析、定位、抽取的基本外壳。

这些都说明大家不是只在手工整理材料，而是在尝试把任务做成可扩展的程序。

### 4. 暴露问题本身也是有价值的

有些结果错误较多，但错误类型很有代表性：关键词误抽、投资者为空、事件重复、证据来自风险提示或战略配售、人工摘要代替原文证据等。这些问题帮助我们更清楚地定义下一版通用抽取方法。

从这个角度看，第一周不是简单评分，而是一次共同探索：我们已经知道哪些做法有效，哪些做法会系统性出错。

## 二、共同需要改进的地方

### 1. “生成 JSON”不等于“抽取成功”

后续抽取建议大家采用一个统一标准：**一条融资事件是否可信，关键不是 JSON 是否存在，而是它能不能回到招股书原文。**

合格事件至少应满足：

- 有明确交易动作，如增资、认购、股权转让、定向发行等。
- 有发行人或发行人前身作为交易主体。
- 有投资者或认购方、受让方。
- 有金额、股数、价格、比例、估值等至少一种数值信号。
- 有原文证据，且不是人工概括；证据可以是 `evidence_text`，也可以是 `evidence_chunk_id` 加单独保存的证据块文件。
- 投资者名称应能在对应证据文本或证据块中找到。

如果这些条件不满足，建议不要直接放入正式 JSON，可以先进入人工复核队列。

### 2. 候选文本定位要先看章节，再看关键词

很多误抽来自一个共同原因：全文看到“投资者”“增资”“融资”就截取。招股书里“投资者”这个词会出现在风险提示、投资者保护、战略配售、发行公告、募集资金用途等很多无关章节里。

后续应优先定位这些章节：

- 发行人基本情况
- 历史沿革
- 股本形成及变化
- 历次增资及股权转让
- 发行人设立及报告期内股本和股东变化
- 新增股东入股情况、入股原因、入股价格及定价依据
- 报告期内发行融资
- 股东私募基金备案情况

同时应过滤或降权这些章节：

- 风险因素
- 重大事项提示
- 本次发行概况
- 战略配售
- 募集资金用途
- 投资者保护
- 股权激励、股份支付
- 现金流量分析
- 客户、供应商、关联交易
- 董监高承诺、锁定期安排

### 3. 事件边界要清楚，不要把关键词片段当事件

一条融资事件通常对应一个小标题、一个表格或一段完整叙述，例如“2020 年 9 月，增资至 13,380.00 万元”。不要把相邻几行关键词碎片分别生成多条事件，也不要把同一事件重复拆成很多条。

建议后续按“小标题到下一个同级标题”或“事件说明加完整表格”来生成事件包，再交给规则或 LLM 抽取。

### 4. LLM 提示词和代码都要提交

如果使用 LLM，需要提交：

- 模型名称。
- system prompt。
- user prompt 模板。
- temperature、max tokens 等参数。
- 输入候选文本。
- 原始输出或抽取日志。
- 校验日志。

否则即使结果看起来不错，后续也很难判断它是模型稳定抽取出来的，还是经过了人工修补。

如果不用 LLM，也可以，但需要说明规则逻辑，并提交足够严格的校验程序。

### 5. 证据应尽量保留原文，不要用摘要替代

证据应该是招股说明书原文片段或可回源的证据块，可以有少量空格清理，但不应改写成自己的话。人工摘要可以放在 `notes` 或 `review_notes`，不要替代原文证据。

这一点非常关键，因为后续所有复核都依赖证据回源。

### 6. 不要用空投资者占位

如果识别不出投资者，不要写：

```json
"investor_original_name": ""
```

这种空投资者会让无效事件看起来像有效事件。更稳妥的做法是把这条记录标记为 `manual_review`，说明“投资者未识别”。

### 7. 校验要比抽取更严格

后续建议每组至少提交一个校验日志，检查：

- JSON 是否能解析。
- 必填字段是否齐全。
- `evidence_text` 或证据块是否能在原文中找到。
- 证据是否含交易动作。
- 证据是否含数值信号。
- 投资者名称是否出现在证据文本或证据块中。
- 是否来自坏章节。
- 是否有重复事件。
- 是否有空投资者。

## 三、按GitHub提交的具体建议

### 1. 刘宇轩、苏宇垚：`https://github.com/flyxx-pro/team-star`

你们这组最大的优点是提交了较完整的 LLM 抽取流程和提示词。提示词里已经明确要求“未披露不编造”“轮次不推断”“投资者原文名称”“每条事件要有 evidence_text”，这是很好的方向。结果中的证据字段整体也比较扎实。

下一步重点改进：

- 补充候选文本文件，否则后续较难判断漏抽或误抽来自定位环节，还是来自 LLM 抽取环节。
- 增加强校验，不只检查字段是否存在，还要检查证据是否回源、投资者名是否出现在证据文本或证据块中、是否来自坏章节。
- 提示词中虽然要求“不推断轮次”，但结果中仍可能出现模型推断痕迹。建议在校验阶段强制把未披露轮次改为“未披露”。
- 保留 LLM 原始输出和失败样本，方便做 bad case 分析。

建议你们下一步目标是：把现有 LLM 流程从“能抽取”升级为“可复核、可复现、可批量运行”。

### 2. 赵秉清：`https://github.com/tdj49vhhwv-wq/prospectus--project.git`

这个仓库的优点是材料丰富，候选文本、章节列表、异常记录和流程文档都比较认真。你对“先目录、再章节、再候选文本、再失败记录”的流程有比较清楚的意识。

下一步重点改进：

- 证据需要改成原文片段或可回源证据块，不要用人工概括替代。概括可以放在备注里。
- schema 要回到任务书要求的结构，尤其是 `financing_events` 建议放在顶层，避免后续合并和校验困难。
- 代码不要写死本机路径，要改成相对路径或配置文件。
- 如果使用 LLM，需要提交可执行 prompt，而不是只在说明文档里写一个示例。
- 目前结果更像“人工整理报告”，下一步要转成“机器可运行、机器可校验”的流水线。
- 需要补齐下载、解析、定位、抽取、校验之间的日志链条，让别人能判断每一步是否成功。

建议这个仓库下一步目标是：把丰富材料压缩成规范 JSON，并把证据从“总结性语言”改成“可回源原文”。

### 3. 杨苗鑫：`https://github.com/yang111-svg/prospectus-pevc-project/pull/1`

这个提交的优点是目录结构和文件覆盖面比较完整，也有 PR 修补意识，说明你在尝试按项目规范推进。但目前抽取规则太浅，很多地方是简单正则命中后直接生成事件，因此误抽和重复较多。

下一步重点改进：

- 不要用“关键词命中”直接生成融资事件。需要先判断是否在正确章节、是否有交易动作、是否有投资者、是否有数值信号。
- 解决 README 中的冲突标记、BOM、`__pycache__`、硬编码 Windows 路径等工程问题。
- 输出 schema 要与任务书一致，投资者要用数组结构，不要只写一个浅层 `investor` 字段。
- 增加去重逻辑，避免同一事件被重复抽成多条。
- 增加坏章节过滤，特别是风险因素、战略配售、募集资金、投资者保护、股权激励。
- 提交前请重新检查 PR diff，确保没有把临时文件、缓存文件、未解决冲突提交上去。

建议这个 PR/仓库下一步目标是：少做一些样本，但把事件边界、证据回源、schema 和去重做准。

### 4. 尚恬：`https://github.com/Shangtian-maker/work`

这个仓库的优点是流程外壳已经搭起来了，字段覆盖也比较多，包括日期、轮次、金额、估值、股份、投资者、退出状态等，也处理了编码兼容问题。

但目前最大问题是入口太宽，导致大量无关文本被当成融资事件。结果中事件数量异常膨胀，很多投资者为空，说明抽取阶段需要先回到候选定位与过滤环节，把入口重新收窄。

下一步重点改进：

- 不要继续使用“没有投资者就补空投资者”的做法。投资者为空的事件不宜进入正式 JSON。
- 先重写候选文本定位，只保留发行人基本情况、历史沿革、历次增资、股东变化、新增股东等核心章节。
- 加入坏章节过滤，删除封面、声明、目录、重大事项提示、风险因素、IPO 战略配售、募集资金用途、现金流、股权激励等文本。
- 事件生成前应同时满足：交易动作、投资者、数值信号、发行人主体、原文证据。
- 校验日志要标出每条事件为什么通过或为什么删除。
- 建议先把少数样本抽准，再逐步扩大样本量。

建议这个仓库下一步目标是：从“广撒网”改成“窄入口、强过滤、少而准”。

下一步重点改进：

## 五、Week 2统一提交要求

本周仍然只提交一个 GitHub 仓库地址。这里说“少文档”，是指老师端不再额外发很多说明材料；学生端仍需要把代码、表格、校验结果和周报放在 GitHub 仓库里，方便复核和比较。

老师给出的 `603418_友升股份_抽取示范.jsonl` 和 `603418_友升股份_三表抽取示范.xlsx` 是最基础的可复核示例，不是上限。各组可以在保留两类基础事实记录、PDF 证据回源、Pydantic schema 和带数字 cross-check 的前提下，补充更完整的字段、更细的证据定位、更严格的校验逻辑或更好的自动化流程；做得更细、更稳，会作为本周的重要加分点。

每组最低提交：

1. `outputs/week2_jsonl/`：8 家公司从 PDF/解析文本中抽取出的 JSONL。JSONL 是主结果，不是 Excel。
2. `outputs/week2_excel/`：由 JSONL 转出的 Excel 查看版，每家公司 1 个 Excel，参考 `603418_友升股份_三表抽取示范.xlsx`。
3. `schemas/`：用 Pydantic 定义 JSON/JSONL 中两类事实记录的字段、类型、必填项和基本取值规则。
4. `scripts/validate_jsonl.py`：读取 8 家 JSONL，运行 schema 检查和数值 cross-check。
5. `scripts/jsonl_to_excel.py`：把 JSONL 转成 Excel，方便人工查看。
6. `logs/schema_validation_log.csv`：记录每家公司、每类记录、每类检查是否通过。
7. `logs/cross_check_summary.csv`：记录总量核对和逐股东核对结果，并保留参与核对的数字。
8. `annotations_pdf/`：保留关键 PDF 批注截图或批注 PDF，用来说明字段来自哪里。
9. `weekly_reports/week2.md`：简短说明本周做了什么、哪些公司已经对上、哪些还需要复核。

可以继续提交 `code/` 和 `prompts/`，但本周评价重点不是代码炫技，而是 JSONL 是否按示范变量定义抽取、证据是否来自 PDF 原文、Pydantic 检查和数值 cross-check 是否能复现。

### JSONL 与 Excel 查看版的统一口径

请参考 `603418_友升股份_三表抽取示范.xlsx`。学生需要从 PDF/解析文本中抽取的是 JSON/JSONL；Excel 只是把 JSONL 转出来后的查看版。

这份示范只定义最低可复核口径。不要低于这个口径，但可以在此基础上做得更完整、更细致。

JSONL 至少包含两类事实记录：

- `subscription_flow`：对应 Excel 第 1 张 sheet `认缴流量`。
- `equity_snapshot`：对应 Excel 第 2 张 sheet `股权结构存量`。

Excel 第 3 张 sheet `schema_cross_check` 是 Pydantic schema 和数值 cross-check 的输出，不是学生从 PDF 手工抽取的第三类事实。

`认缴流量` 回答：谁在什么时候认购了多少、多少钱、什么价格。

- `PDF页码`
- `增资日期`
- `认购方`
- `认购数量(万股)`
- `认购金额(万元)`
- `认购价格(元/股)`
- `原文证据`

`股权结构存量` 回答：某个时点股东结构是什么。必须有 `t0`，也就是报告期初或可识别的最早股权结构。

- `PDF页码`
- `时点`
- `股权结构口径`
- `总股本(万股)`
- `总出资额(万元注册资本)`
- `股东名称`
- `持股数(万股)`
- `出资额(万元注册资本)`
- `持股比例`
- `原文证据`

如果 PDF 只披露出资额，就只填出资额列，持股数先空着；如果 PDF 只披露持股数，就只填持股数列，出资额先空着。不要为了填满表格去倒推。

### Pydantic schema 与 cross-check

第三步不是再手填一张事实表，而是用 Pydantic schema 和校验脚本检查 JSONL。至少要检查：

- JSONL 是否能逐行解析。
- 每行 `record_type` 是否属于 `subscription_flow` 或 `equity_snapshot`。
- 转成 Excel 后，前两张事实表的列名是否完全匹配示范。
- `PDF页码`、`增资日期`、`认购方`、`股东名称`、`原文证据` 等必填字段是否为空。
- `认购数量(万股)`、`认购金额(万元)`、`认购价格(元/股)`、`持股数(万股)`、`出资额(万元注册资本)` 等数字列是否能解析为数值。
- 每家公司是否存在 `t0` 股权结构。
- 同一时点的股东持股数合计是否等于该时点披露的 `总股本(万股)`；如果 PDF 只披露出资额，则先检查出资额合计。
- 每次增资的认购数量合计，能否对应相邻存量时点的股本变化。
- 每个新增认购方在增资后存量表中的持股数，是否等于认缴流量表中的认购数量。
- 老股东持股是否保持、稀释、退出，或存在需要人工复核的未解释变化。

校验结果写入 `logs/schema_validation_log.csv` 和 `logs/cross_check_summary.csv`，并可转成 Excel 第 3 张 sheet 查看，不要靠口头说明。`cross_check_summary.csv` 和 Excel 第 3 张 sheet 不能只写 `pass/fail`，至少要保留以下数字列：

- `上一时点股本/持股数(万股)`
- `上一时点出资额(万元注册资本)`
- `本次认缴/变化(万股)`
- `预期变更后股本/持股数(万股)`
- `PDF披露变更后股本/持股数(万股)`
- `差额(万股)`

## 六、Week 2评价重点

本周重点不是多抽 PE/VC 事件，而是先把上市前股本变化理清楚。

评价时主要看：

- 8 家公司是否完整，不得漏做、错做或继续使用旧样本。
- 每家公司是否找到 `t0` 股权结构。
- 认缴流量表和股权结构存量表是否分开。
- 数字列是否只填 PDF 直接披露或能明确定位的数值，单位是否写在列名中。
- 持股数和出资额是否分列，PDF 未直接披露的字段是否留空。
- `原文证据` 是否能回到 PDF 页码和原文表述。
- 是否提交 Pydantic schema，并能检查 JSONL 的 record_type、字段类型和必填字段。
- 是否做了总量 cross-check：本次认缴数量合计能否对应股本变化，并在日志中展示参与核对的数字。
- 是否做了逐股东 cross-check：每个新增认购方、老股东变化后持股能否逐项对上，并在日志中展示上一时点、本次变化、预期值、PDF值和差额。
- 对不上时是否在校验日志中明确标为 `待复核`，而不是硬凑数。

## 七、总结

第一周说明大家已经能跑下载、解析和初步抽取，但也说明“直接抽 PE/VC 事件”太容易黑箱化。第二周先退一步：把 8 家公司的上市前股本变化做清楚，再在这个基础上讨论哪些投资者属于 PE/VC。

本周最重要的标准只有一句话：**每一个数字都要能回到 PDF，每一次变化都要能通过 Pydantic schema 和数值 cross-check 复核。**
