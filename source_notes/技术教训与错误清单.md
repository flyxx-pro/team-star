# 技术教训与错误清单

> 三周实战总结。每一个教训背后都有踩坑的代价。

---

## 一、PDF解析

### 错误1：MinerU 页码范围随便填
**后果**：三协电机第一份JSON 200页全是财务会计附录，"增资"仅1次，"发行人基本情况"0次。三联锻造第一份JSON"增资"0次。20多次上传，报废率约50%。

**正确做法**：
1. 先看PDF总页数
2. TOC定位找到"发行人基本情况"起始页
3. 页码范围 = 起始页 ~ 下一节起始页
4. 提取前验证关键词："增资" ≥ 3次？"股权转让" ≥ 2次？

### 错误2：只用JSON模式，不用Markdown模式
**后果**：星图测控第47页表格MinerU检测到了框架但单元格内容全为空。三次JSON导出全部失败。

**正确做法**：
- JSON模式 → 段落文字 + 页码标记（喂AI）
- Markdown模式 → HTML表格 → 正则直接解析（0 token）
- 双模各导出一次，互补

### 错误3：忘记PDF附件
**后果**：三联锻造主PDF只说"详见申报文件4-3"，四次股权转让和四次增资的详细数据漏掉。AI只能抽到1次增资，实际有多次。

**正确做法**：读PDF时注意"详见附件""见申报文件XX"等引用，附件内容单独上传MinerU。

---

## 二、AI提取

### 错误4：全文输入AI，不定位章节
**后果**：招股书里"投资者"出现在风险提示、募集资金用途、战略配售等无关章节。全文输入导致AI误判，把风险提示里的"投资者"当作股东。

**正确做法**：TOC定位 → 提取"发行人基本情况"章节 → 只把这部分喂给AI。输入文本缩小90%，准确度反而更高。

### 错误5：表格走AI，不走正则
**后果**：MinerU展平表格后行列关系丢失，AI面对"中科星图3825000046.36%34.77%"这种连续字符串，无法区分哪个数字是什么。

**正确做法**：表格数据走Markdown → 正则 → 0 token → 100%准确。AI只做段落文字的语义理解。

### 错误6：不验证，直接交
**后果**：第二周被老师批评"字段有缺失""没有和教师版本比对""提交规范但结果质量不足"。友升2022年10个认购方数值全为null自己没发现。

**正确做法**：每次提取后跑Cross-check。持股比例合计=100%？总股本前后一致？有null值？有重复？任何一个不通过就回PDF查。

---

## 三、数据校验

### 错误7：只做总量校验，不做比例校验
**后果**：星图测控7500+750=8250通过了，但持股比例只加了55.45%。44.55%的数据缺口靠手动加总才发现。如果没有这个习惯，交了也不知道自己漏了一半数据。

**正确做法**：Cross-check必须含持股比例合计检查。偏离超过1%自动标记，不等就是有缺口。

### 错误8：gold和auto混在一起
**后果**：老师指出"teacher_gold、manual_gold、toc_located、auto混在一起，外部无法判断哪些是人工确认的"。看起来数据量大，但可信度低。

**正确做法**：
- `manual_gold/` → 只放人工核对过PDF原文的
- `auto_output/` → 只放机器自动跑的
- `comparison/` → 放两者差异
- 每条记录标注 `source` 和 `review_status`

## 四、Prompt设计

### 错误9：Prompt不分披露类型
**后果**：对段落式披露用"重建行列关系"Prompt→胡乱均分（三协电机265万的来历）。对表格式披露用"不要均分"Prompt→漏掉所有数值。

**正确做法**：先判断披露类型——段落还是表格——再选对应Prompt。段落：告诉AI"不要均分，个体未披露就填null"。表格：告诉AI"从展平文本中根据数字上下文重建行列关系"。

### 错误10：Temperature设太高
**后果**：早期Temperature=0.3时AI有"创造性"，会擅自调整数字、调换顺序。

**正确做法**：数据提取任务Temperature=0.0或0.1，最大化确定性。不要给AI"发挥"的空间。

## 五、流程设计

### 错误11：一次性处理8家，没先跑通1家
**后果**：三周都在反复修同一类问题。如果第一周先跑通友升股份的完整闭环——从PDF到Excel每一步验证对——再扩展到8家，效率会高很多。

**正确做法**：先选1家最规范的公司（如友升），跑通完整闭环，验证每一步。再扩展到其他7家。

### 错误12：出了问题先调Prompt，不看PDF原文
**后果**：三联锻造AI说只有1次增资，反复调Prompt无效。翻到PDF第35页才发现"详见申报文件4-3"。浪费了大量时间在Prompt上调来调去。

**正确做法**：AI结果不对 → 第一步不是调Prompt → 先翻PDF原文 → 确认是输入问题还是Prompt问题。大多数情况下输入材料本身就不完整。

## 五、Markdown/HTML解析

### 错误13：HTML表格有 rowspan/colspan，正则解析漏掉整张表
**后果**：赛分科技 2021-11-10 增资有 10 个认购方，Markdown 第 854 行有完整 HTML 表格（认购股份/价格/金额全齐）。但价格列用了 `rowspan="10"`（10 人同价 127.24 元/股），导致后续 9 行的单元格位置偏移——原以为 cells[3] 是价格实际是 cells[2]。加上列名是"认购股份(股)"而非"认购数量(股)"，列名匹配也失败了。双重原因导致整张表 10 个认购方全部漏掉。误判为"赛分科技认购明细 Markdown 未覆盖"，实际数据一直在 Markdown 里。

**正确做法**：
1. 表格解析前先扫描 `rowspan` 和 `colspan` 属性，有则展开为完整行列后再解析
2. 列名匹配用模糊正则：`认购.*[股数]` 而非精确匹配 `认购数量(万股)`；`金额|万元|元` 而非精确匹配 `认购金额(万元)`
3. 提取后做完整性检查：已知的增资轮次是否都有对应的认缴记录？缺轮次大概率是 rowspan 导致整张表被跳过
4. **不轻易下"Markdown 没数据"的结论——先检查 HTML 源码，确认表格是否被 rowspan/colspan 破坏了列结构**

---

## 总结：每个环节的正确顺序

```
收到结果不对
  ↓
① 翻PDF原文 → 确认数字在哪页、怎么写的
  ↓
② 检查MinerU覆盖 → 关键词≥3次？页码范围对吗？
  ↓
③ 检查是否是人做不到的事 → 表格丢了？→ Markdown补
  ↓
④ 才调Prompt → 段落/表格用不同Prompt
  ↓
⑤ 跑Cross-check → 比例对？总股本对？
  ↓
⑥ 不确定的标review_queue → 不是硬填正确
```

**核心原则：理解PDF原文 > 调Prompt。PDF里找不到的数字，AI不可能凭空变出来。**
