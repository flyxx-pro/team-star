# Week 3 Error Analysis — 刘宇轩 & 苏宇垚

## 1. 哪些事件可以规则稳定提取

| 事件类型 | 可规则提取 | 说明 |
|---------|-----------|------|
| 增资日期 | 部分 | 正则可匹配"YYYY年MM月DD日"格式，但同一页可能有多轮增资需人工区分 |
| 认购方名称 | 部分 | 表格中的股东名可用规则提取，但自然段落中的需LLM |
| 认购数量/金额 | 否 | 单位不统一（万元/元/美元），需LLM做单位归一化 |
| 总股本变化 | 部分 | 可用正则匹配"增至XXX万元"模式 |
| 章节定位 | 是 | 关键词"历史沿革""股本形成"定位章节，规则可稳定完成 |
| 时点标签(t0/t1) | 否 | 需理解事件语义，规则无法自动判断哪些是关键时点 |

## 2. 哪些事件必须依赖LLM或人工

- **表格中的数值抽取**：认购明细表、股权结构表 → 需LLM理解表结构
- **单位判断**：PDF中"股""万股""元""万元""美元"混用 → 需LLM归一化
- **事件类型分类**：增资 vs 股权转让 vs 整体变更 vs 资本公积转增 → 需LLM
- **t0判定**：哪个是最早可识别的股权结构 → 需人工确认

## 5. 单位错误如何发现

- Schema校验中检查数值范围：持股比例>100立即报警
- Cross-check中：同一时点持股比例合计≠100%说明单位或数值有误
- 人工比对：与教师示范Excel的数值量级比对（股vs万股差10000倍）

## 6. 股权转让如何与增资分开

- Prompt中明确要求：只抽增资/认购，不抽股权转让
- 新增 share_transfer_flow 类别
- 规则辅助：股权转让的特征是"转让方+受让方"，增资的特征是"认购方+新增注册资本"

## 7. Cross-check失败后流程如何处理

1. 标记为 manual_review_queue
2. 回到PDF原文找证据
3. 判断是抽取错误还是PDF未披露
4. 抽取错误→修Prompt重试
5. PDF未披露→标记为null，记录原因

## 8. 扩展到50家公司最先失败在哪

- **PDF解析失败**（三联锻造案例）：MinerU对扫描版或加密PDF完全无法解析。50家中预计5-10%会有此问题，需OCR或人工摘录
- **章节标题不统一**：北交所/创业板/主板的招股书章节标题格式不同。例如北交所常用"发行人基本情况"而非"第五节 发行人基本情况"
- **单位混用**：同一份招股书中"股"和"万股"、"元"和"万元"可能混用。赛分科技的认购明细就没有显式的"认购数量"和"认购金额"列标题
- **Deepseek API成本**：以每家公司3次API调用（认缴+存量+转让），50家=150次调用，每次约28K tokens输入，总成本约$15-20
- **MinerU并行瓶颈**：目前为手动网页上传，50家需要数小时人工操作。需切换到API批量调用

## 9. 已知错误与修正计划

| 错误类型 | 例子 | 修正 |
