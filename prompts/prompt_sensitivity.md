# Prompt 敏感性分析

## 测试设置

- 模型：Deepseek-chat（固定）
- Temperature：0.1（固定）
- 测试公司：赛分科技 (688758) —— 数据量大，差异明显
- 输入文本：相同（MinerU 新 JSON，86,582字符，密度评分压缩至 40,575字符）

## 对比的 Prompt 版本

### Prompt A（原始完整版，来自 extract_week2.py）

特点：约 800 字，详细说明字段定义、单位、排除规则、示例格式。包含"未披露不编造""每个认购方单独一行""单位已在列名中写明"等约束。

```
关键约束：
- 只抽增资/认购事件，不抽股权转让
- 金额和股数只填PDF明确披露的数值，不要推算
- 单位已在列名中写明（万股、万元），数值不要带单位
- 每一笔认购方单独一行
- 必须包含t0
```

### Prompt B（精简版，来自赛分专项修复）

特点：约 200 字，聚焦关键要点，增加了单位转换提示。

```
关键约束：
- 认购明细经常在表格中，即使没有列标题也要从上下文提取数值
- 单位转换：如是"股"除以10000，如是"元"除以10000
```

## 效果对比

| 指标 | Prompt A | Prompt B |
|------|----------|----------|
| 认缴流量 | 0 条（JSON截断） | 15 条 |
| 股权存量 | 0 条（JSON截断） | 153 条 |
| null值 | N/A | 14/15 |
| JSON解析成功 | 否（输出超长） | 是 |
| 优点 | 字段定义完整，约束明确 | 输出更紧凑，不会截断 |
| 缺点 | 输出过于详细，导致JSON截断 | 数值提取能力弱，14/15为null |

## 最终方案（Prompt A+B 组合）

赛分科技最终采用**分两轮调用**策略：
1. 第一轮：单独提取 subscription_flow
2. 第二轮：单独提取 equity_snapshot

这避免了单次输出超长导致的 JSON 截断问题。最终结果：15认缴 + 153存量。

## 关键发现

1. **输出长度是瓶颈**：数据量大的公司（如赛分、云汉），完整 Prompt 会导致 JSON 超过 max_tokens 而截断
2. **分轮调用解决截断**：subscription_flow 和 equity_snapshot 分开提取，各走各的 token 预算
3. **null 值不是 Prompt 问题**：无论哪种 Prompt，认购数量/金额的 null 率相似，根因是 MinerU 文本中表格数据被展平，AI 无法定位"哪一行对应哪个认购方"
4. **Temperature=0.1 稳定**：两次同 Prompt 运行结果接近（15 vs 14 events），偏差在 1-2 条以内

## 建议

- 数据量小的公司（< 10 events）：用 Prompt A 一次提取
- 数据量大的公司（> 10 events）：用 Prompt A + 分轮调用
- null 值的最终解决不靠 Prompt 优化，靠 MinerU 输出更好的表格结构、或人工复核补充
