# 会议总结 — 刘宇轩

## 一、问题回顾

本次会议中，我回顾了三周实践暴露的三个核心问题。

第一，MinerU 解析遗漏附件内容。以三联锻造为例，最初只上传了招股书主 PDF，AI 仅提取到 1 条认缴记录。事后翻到 PDF 第 35 页才发现原文写明"四次股权转让和四次增资详见申报文件 4-3"——主 PDF 只写了一行引用，详细数据全在附件里。补传附件后，认缴从 1 条增至 12 条，存量从 4 条增至 28 条。这个教训说明：MinerU 的解析范围直接决定了数据完整性的上限，上传前必须确认是否遗漏了申报文件附录。

第二，JSON 格式解析后表格被展平，行列对应关系丢失（已通过 Markdown 模式解决）。PDF 中的表格原本是规整的行列结构——第一列股东名称、第二列持股数量、第三列持股比例。MinerU JSON 模式将其转为纯文本后，"中科星图 38,250,000 46.36% 34.77%"这样的连续字符串让 AI 无法区分哪个数字属于哪个字段。星图测控第 47 页的股权结构表在三次 JSON 导出中均无法提取单元格内容。第四周全面切换为 MinerU Markdown 导出模式后，HTML `<table>` 标签完整保留了表格行列结构，编写正则解析脚本（code/07_markdown_pipeline/parse_tables_to_jsonl.py）成功提取全部 7 个股东数据，持股合计 8250 万股、比例合计 100%，全程零 token 消耗。该方案已推广至全部 8 家公司的 Markdown 文件。

第三，不同公司的 PDF 披露格式存在差异。三协电机的定向发行只披露了总发行量（"向稳正景明和长泽创投定向发行 530 万股"），未逐人拆分各自认购数额；黄山谷捷是有限责任公司，股权转让用的是出资额转让表述而非标准的股份转让格式。格式不统一导致 AI 出现误判——对段落式披露擅自分摊、对出资额格式无法识别事件类型。

## 二、老师反馈

老师提出了三个方向的指导。

一是建立自己的知识体系框架。核心转变是从"全流程依赖 AI 决定怎么做"变成"人掌握框架和方向，AI 负责执行具体步骤"。人制定提取策略、设计校验规则、判断数据合理性；AI 执行文本抽取、格式转换、字段填充。从 AI 做计划、人被动接受，变成人做架构、AI 做执行。

二是遇到不懂的问题主动学习。不要只在大模型里反复试错，应该去小红书、B 站、技术社区检索相关知识和教程，拓宽解决问题的路径。

三是技术方法上的具体建议。不要把整本 PDF 直接扔给 AI，应该教 AI 怎么检索——先通过目录定位章节、再精准提取相关文本，既节省 token 又提高准确度。针对表格展平的问题，直接用 MinerU 的 Markdown 模式导出，既能保留表格结构，又不需要消耗 token。

## 三、会议收获

本次会议最大的收获是认知转变：从被动的任务推进者变成主动的知识学习者。

三周实践中，我一度陷入反复调 Prompt 的循环——AI 结果不对就改参数、加约束、换模型，但很少停下来问自己：这一步到底在做什么？为什么表格提取失败？是 Prompt 的问题还是输入材料本身就缺失了数据？

老师的反馈让我意识到，真正有效的路径是：先理解每一步运作的内在逻辑和含义——TOC 定位为什么能省 token、表格展平为什么 Prompt 解决不了、Cross-check 为什么比 AI 自己的输出更可靠——然后才谈得上"更好地运用大模型"。不理解业务逻辑和数据结构，有再好的工具也只是在黑箱外面反复敲打。

后续我会把这条认知落实到实践中：遇到问题先翻 PDF 原文理解数据，再判断是输入问题、工具问题还是 Prompt 问题，最后才动手修改。技术工具是手段，理解业务才是基础。

## 四、后续推进

会议结束后，我将上述认知转化为四个具体改进。

第一，将 MinerU 导出格式从 JSON 全面切换为 Markdown，并在星图测控上完成了完整验证。JSON 模式下表格数据几乎全部丢失——星图测控第 47 页的股权结构表三次导出均为空，最多只能提取到 3 个股东、持股合计仅覆盖 44.55%。切换 Markdown 后，表格以 HTML `<table>` 标签完整保留，用正则直接从完整 Markdown 文件中解析出"发行前后公司的股本情况"全量表格，成功提取全部 7 个股东，持股合计 8250 万股、比例合计 100%，与 PDF 原文完全一致。同时从定向发行认购表中提取了 3 条认缴记录（策星银河 279 万股、策星逐日 171 万股、幸福二期 300 万股），认购合计 750 万股，与存量变化量互相验证通过。全程表格提取零 token 消耗。后续流程确定为：段落文字走 JSON 喂 AI、表格数据走 Markdown 正则解析，两者互补。

第二，完善了 AI 提取的技术路径，针对不同公司类型的口径差异提前配置 Prompt 和 Schema。有限责任公司和股份有限公司在出资记录上存在根本性差异——前者用"出资额（万元注册资本）"、后者用"持股数（万股）"；前者的股权转让用出资额转让格式、后者用股份转让格式。如果不将这两类差异提前写入 Prompt 告知 AI，就会导致黄山谷捷的股权转让无法识别、三联锻造的认购数量字段全部为 null。会议后我将公司类型判断加入预处理流程，在调用 AI 前先识别公司性质并匹配对应 Prompt。

第三，完善了大模型的提取流程——先通过 TOC 目录定位章节、再精准提取相关文本。之前的问题是全文输入或凭感觉设页码范围，既浪费 token 又引入无关内容的噪声。改为先扫描目录页找到"发行人基本情况"的起始页和结束边界，只从这个范围提取文本，输入量缩减约 90%，准确度反而更高。这一优化已写入标准化流程文档。

第四，进一步完善了精讲公司的 PPT。以星图测控为深讲案例，在回到 PDF 原文逐页核对数据的基础上——包括第 35 页的定向发行段落、第 47 页的发行前后股本对比表、第 48 页的历史代持记录——结合大模型提取的数据结果和手工 cross-check 验证，制作了 7 页 PPT，覆盖了老师要求的四个方向：定向发行、员工持股平台、发行前股东结构对应关系、流量存量 cross-check。

第五，通过上述优化，单家公司的 Token 消耗实现了数量级下降。对比前后效率：

| 环节 | 旧方式 | 新方式 | 节省 |
|------|--------|--------|------|
| 章节定位 | 全文输入 AI（~45000 tokens） | TOC 目录定位（0 token） | 100% |
| 表格提取 | AI 从展平文本中抽取（~5000 tokens/次，且常失败） | Markdown 正则解析（0 token） | 100% |
| 段落文字提取 | 全量文本输入（~45000 tokens） | 精准章节截取（~6000 tokens） | 87% |
| **单家公司合计** | **~50000 tokens** | **~6000 tokens** | **节省约 88%** |

以星图测控为例，旧方式单次提取约消耗 45000-50000 tokens（约 ¥0.09），其中表格部分占用大量 token 且反复失败。新方式表格走零 token 正则、段落走 6000 token AI 调用，单家公司降至约 ¥0.012。扩展到 50 家公司的预估成本从旧方式的 ¥4.5 降至 ¥0.6，节省约 87%。更重要的是表格数据从"经常丢失"变成了"100% 完整"——这不是成本优化，是数据质量的根本性提升。
